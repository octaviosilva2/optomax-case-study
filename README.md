## English

**B2B SaaS for clinical management in optometry practices** — scheduling, patient
records, digital prescriptions (PDF) and recurring billing. Multi-tenant: every
clinic is isolated by `org_id`, enforced with Postgres Row Level Security.

In production since May 2026. This repo is a sanitized, history-free snapshot published for portfolio purposes — see [note below](#a-note-on-this-repository).

### The problem

Optometry clinics mostly run on spreadsheets, paper agendas or generic scheduling
tools. None of those cover the domain-specific parts of the workflow — the optical
prescription (sphere, cylinder, axis, addition), tracking a patient's grade evolution
over time, or a billing flow built for a subscription business. OptoMax is a vertical
SaaS built to close that gap end-to-end: schedule → clinical record → prescription →
recurring payment.

### What it does

- Multi-tenant scheduling and patient records, isolated per clinic (`org_id` + RLS)
- Digital clinical record with visit history and grade evolution over time
- Optical prescriptions rendered to PDF, with signature capture
- Recurring subscription billing (Pix + credit card) via ASAAS, driven by an
  idempotent webhook
- Admin back-office with independent authentication (HMAC-signed session +
  mandatory TOTP 2FA)
- LGPD-aware handling of health data — documented Terms of Use / Privacy Policy,
  data-processor role declared

### Tech stack

- **Next.js 16** (App Router) + **React 19** + **TypeScript 5**
- **Tailwind CSS 4** + **shadcn/ui** (base `@base-ui/react`) + `next-themes`
- **Supabase** — Postgres, Auth, Storage, RLS (`@supabase/ssr` + `@supabase/supabase-js`)
- **TanStack Query 5** (server state) · **react-hook-form 7** + **zod 4** (forms)
- **@react-pdf/renderer** (prescriptions) · **recharts** (charts) · **sonner** (toasts)
- **ASAAS** (Pix + recurring card) · **Resend** (transactional email) · **Sentry** (observability)
- **Testing:** Vitest (unit) · Playwright + `@axe-core/playwright` (e2e/a11y) · Lighthouse CI

### Architecture highlights

- **Multi-tenant by RLS, not by application logic.** Every table has Row Level
  Security active; `org_id` always comes from the authenticated session, never
  from the request body.
- **Server Actions only for mutations**, validated end-to-end with `zod`.
- **Schema changes only via versioned migrations** (`supabase/migrations/`) —
  never applied by hand in production.
- **The payments webhook is the single writer of subscription state**, applied
  idempotently — fixed after an earlier gap where checkout alone couldn't persist it.
- **CI gate on every PR to `main`:** type-check (`tsc --noEmit`) + unit tests
  (Vitest) + production build must pass before merge.
- **Daily automated database backups to external storage**, outside the primary
  vendor, with a **weekly automated restore drill** that restores into a disposable
  database and checks row counts — proof the backup restores, not just that it exists.
- **Separate, hardened auth for the internal back-office:** HMAC-signed session
  cookie + mandatory TOTP 2FA, fails closed if misconfigured.

### Case study — building this with Claude Code

**Context.** OptoMax needed to go from idea to a production SaaS handling real
money and real patient data, fast, without an engineering team — the classic
constraint of a two-person founding team where only one side is technical.

**AI-Workflow.** Built through a structured multi-stage agent pipeline
(`dev-agents`), not single-shot prompting: **research → user story → technical
spec → visual gate → backend → frontend → tests → independent validation**, with
three human approval gates (story, spec, final validation) before code lands.
Before any UI work starts, a static before/after HTML mockup is generated for
visual approval — cheap to review, no need to run the app. Handoffs between
sessions are written down explicitly (what's done, what's next, where the
artifacts live) so a "cold" session can resume without re-explaining context.
Run end-to-end via a custom `/dev-agents:feature` skill, or step-by-step when
the approach needs more thought first.

**Architecture.** See [Tech stack](#tech-stack) and
[Architecture highlights](#architecture-highlights) above — decisions made and
revisited over ~119 commits and 37 schema migrations: multi-tenant RLS from day
one, Server-Actions-only mutations, a dedicated subscription write path fixed
after discovering a gap between checkout and webhook, mandatory 2FA added to the
admin back-office, and a backup/restore drill built and *proven* — not just
configured — after the hosting tier ruled out relying solely on the primary
vendor's native protection.

**Evidence.** In production and used by real paying customers since May 2026.
CI (type-check + unit tests + build) gates every merge to `main`. Automated test
suite (unit + e2e + accessibility) plus a weekly job that restores the latest
backup into a disposable database and verifies it. Documented incident runbook,
and LGPD-aware privacy/terms covering health-data processing, built in
partnership with an optometrist co-founder who owns the clinical-domain expertise.

## Português

**SaaS B2B de gestão clínica para optometria** — agenda, cadastro de pacientes,
ficha clínica, prescrições em PDF e evolução do grau. Multi-tenant: cada clínica
é isolada por `org_id`, com Row Level Security no Postgres.

Em produção desde maio de 2026. Este repositório é um snapshot sanitizado e
sem histórico, publicado para fins de portfólio — ver
[nota abaixo](#nota-sobre-este-repositório).

### O problema

Clínicas de optometria em geral operam com planilhas, agenda de papel ou
ferramentas genéricas de agendamento. Nenhuma delas cobre a parte específica do
domínio — a receita óptica (esférico, cilíndrico, eixo, adição), o
acompanhamento da evolução do grau do paciente ao longo do tempo, ou um fluxo
de cobrança pensado para um negócio de assinatura. O OptoMax é um SaaS vertical
construído para fechar essa lacuna de ponta a ponta: agenda → ficha clínica →
prescrição → cobrança recorrente.

### O que faz

- Agenda e cadastro de pacientes multi-tenant, isolados por clínica (`org_id` + RLS)
- Ficha clínica digital com histórico de atendimentos e evolução do grau
- Prescrições ópticas em PDF, com captura de assinatura
- Cobrança recorrente (Pix + cartão) via ASAAS, processada por webhook idempotente
- Back-office administrativo com autenticação própria (sessão assinada por HMAC
  + 2FA/TOTP obrigatório)
- Tratamento de dado de saúde com atenção à LGPD — Termos de Uso e Política de
  Privacidade documentados, papel de operador de dados declarado

### Stack

- **Next.js 16** (App Router) + **React 19** + **TypeScript 5**
- **Tailwind CSS 4** + **shadcn/ui** (base `@base-ui/react`) + `next-themes`
- **Supabase** — Postgres, Auth, Storage e RLS (`@supabase/ssr` + `@supabase/supabase-js`)
- **TanStack Query 5** (estado de servidor) · **react-hook-form 7** + **zod 4** (forms)
- **@react-pdf/renderer** (prescrições) · **recharts** (gráficos) · **sonner** (toasts)
- **ASAAS** — pagamento (Pix + cartão recorrente)
- **Resend** — e-mail transacional · **Sentry** — observabilidade
- **Testes:** Vitest (unit) · Playwright + `@axe-core/playwright` (e2e/a11y) · Lighthouse CI

### Destaques de arquitetura

- **Multi-tenant por RLS, não por lógica de aplicação.** Toda tabela tem Row
  Level Security ativo; `org_id` vem sempre da sessão autenticada, nunca do
  corpo da requisição.
- **Sempre Server Actions para mutação**, validadas ponta a ponta com `zod`.
- **Schema muda só por migration versionada** (`supabase/migrations/`) — nunca
  à mão em produção.
- **O webhook de pagamento é o único escritor do estado de assinatura**,
  aplicado de forma idempotente — corrigido depois de identificar uma lacuna em
  que o checkout sozinho não conseguia persistir esse dado.
- **Gate de CI em todo PR para `main`:** checagem de tipos (`tsc --noEmit`) +
  testes unitários (Vitest) + build de produção precisam passar antes do merge.
- **Backup diário automático do banco em storage externo**, fora do fornecedor
  principal, com **teste de restauração semanal automatizado** que restaura num
  banco descartável e confere a contagem de linhas — prova de que o backup
  restaura, não só que ele existe.
- **Autenticação separada e reforçada para o back-office interno:** cookie de
  sessão assinado por HMAC + 2FA/TOTP obrigatório, falha fechada se mal configurado.

### Regras invioláveis do código

1. **Nunca** importar `lib/supabase/admin.ts` fora de `src/app/api/`.
2. `org_id` vem **sempre da sessão autenticada**, nunca do body da requisição.
3. **Sempre** `react-hook-form` + `zod` em formulário.
4. **Sempre** Server Actions para mutação no banco.
5. Toda tabela tem **RLS** ativo.
6. Schema muda **só por migration versionada** em `supabase/migrations/` (nunca à mão na prod).

### Case study — construção com Claude Code

cobrindo o tratamento de dado de saúde, construídos em parceria com um sócio
optometrista responsável pelo domínio clínico.
